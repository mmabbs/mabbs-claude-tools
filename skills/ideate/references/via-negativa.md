# Via Negativa ("Subtractive Thinking")

**Core operation:** Instead of asking "what should I add?", ask "what should I remove?" Define the solution by elimination. The highest-leverage move is often to stop doing something, not start something new.

## Why It Works

We have an innate addition bias — when asked to improve something, people overwhelmingly add rather than subtract. But removal is often more powerful: fewer dependencies, less maintenance, less cognitive load, fewer failure modes. Nassim Taleb calls this "via negativa" — knowledge of what *not* to do is more robust than knowledge of what to do.

## Process

1. **List everything currently in play** — Features, processes, commitments, dependencies, habits
2. **For each item, ask:** "If this didn't exist, would I add it today?"
3. **Identify removal candidates** — Things you'd never add if starting fresh
4. **Test removal** — What breaks if you remove it? Often less than expected
5. **Remove and observe** — The system usually adapts

## The Subtraction Test

| Question | If Yes... |
|----------|-----------|
| Would I add this today if it didn't exist? | Keep it |
| Does this exist because of inertia? | Remove candidate |
| Does this create more problems than it solves? | Remove candidate |
| Am I maintaining this out of guilt or obligation? | Remove candidate |
| Would anyone notice if this disappeared? | Test removing it |

## Applications

| Context | Subtraction Question |
|---------|---------------------|
| **Product** | Which features do users ignore? What if we removed them? |
| **Process** | Which meetings/reports/approvals add no value? |
| **Code** | Which abstractions create more complexity than they save? |
| **Life** | Which commitments drain energy without returning value? |
| **Writing** | Which sections/paragraphs can be cut without losing meaning? |
| **Strategy** | Which goals are we pursuing out of inertia? |

## Red Flags You Need This

- "It's getting too complex"
- "We keep adding things but nothing gets simpler"
- Feature creep, scope creep, commitment creep
- Feeling overwhelmed by the volume of things to manage
- The answer to every problem is "add another thing"
- Burnout from maintaining too many systems/commitments

## Subtraction Prompts

- "What would I stop doing if I could only keep three things?"
- "What am I maintaining that nobody uses or values?"
- "What would happen if I just... didn't do this?"
- "What's the simplest version that still works?"
- "What am I doing out of habit rather than purpose?"

## Distinction from `simplify`

`simplify` (Upstream Unification) finds one root cause that eliminates many symptoms — it's about finding **better abstractions**. `subtract` is more direct — it's about **removing things entirely**, not replacing them with something better. Sometimes the best abstraction is nothing at all.

## Remember

- Removal is underrated because it feels like giving up — reframe it as *focus*
- "Hell yes or no" — if something isn't clearly valuable, it's a removal candidate
- Subtraction compounds: removing one thing often makes other removals possible
- Pair with `axiom` to rebuild from first principles after clearing the slate
- The goal isn't minimalism for its own sake — it's making room for what matters
